### 所感

**良かった点**:
- PC は全体的にさらに改善（平均 90.8 → 95.7、works/001 と works/004 が 99/98 点）
- モバイル `/works/` が 60 → 73 / FCP 6.6 → 2.4 秒（フォント非同期化が効いた）
- モバイル `/abashiri/works/001/` が 60 → 81 / LCP 7.7 → 4.5 秒（最大改善）

**期待ほど効かなかった点**:
- モバイル LCP は全体平均で 8.3 → 7.7 秒（-0.6 秒）。目標 2.5 秒以下にはほど遠い
- works/001-004 や about のモバイルはほぼ変化なし
- 主因 1: **LCP の主役は画像表示**で、フォント非同期化だけでは届かない
- 主因 2: Xserver の nginx が `.htaccess` の `Cache-Control: max-age=31536000` を無視して 7 日キャッシュに上書きしている（curl 検証済み）

**新たに発生した課題**:
- ⚠️ `/works/001/` モバイル **CLS 0 → 0.176** に悪化（フォントスワップで text reflow が起きている可能性）
- CLS 0.176 は Web Vitals「要改善」域。**要対応**

### 次の打ち手の判断

| 打ち手 | 期待効果 | リスク |
|---|---|---|
| Phase 4 (icons.css 軽量化 52KB→10-15KB) | CSS のレンダーブロック削減 → LCP 直接効く | 中（アイコン漏れ可能性） |
| Phase 5 (Tailwind purge 21KB→5-10KB) | 軽微 | 中 |
| 画像の preload tag 追加（モバイル LCP の本丸） | **大**（LCP の主因への直接打撃） | 小 |
| CLS 悪化対応（works/001 フォントスワップ） | 1 ページだけ Web Vitals 緑域に戻す | 小 |

→ 次セッションで Phase 4 + 画像 preload + CLS 修正に着手するのが効果順位的に最適。

### 主要な改善余地（PageSpeed Insights の「改善できる項目」）

works/004 モバイルで判明済みの項目:

- [ ] 使用していない CSS の削減（89 KiB）→ icons.css と tailwind.css が主因
- [ ] 効率的なキャッシュ保存期間を使用する（16 KiB）→ .htaccess 未設定
- [ ] 次世代フォーマットでの画像の配信 → FV 画像 WebP 化未実施
- [ ] レンダリングを妨げるリソースの除外 → Google Fonts 同期読込

**preload tag の効果は確認できた**:

### 結論

**ここでパフォーマンス改修を一旦終了**と判断。理由:
1. preload tag・display=optional・Phase 1+2 で打てる手は打った
2. PSI の数値ばらつきが大きく、追求する効率が悪い
3. Phase 4+5（icons.css 軽量化・Tailwind purge）は工数大・効果小
4. SEO・顧客行動への ROI は施工事例追加のほうが大きい
5. PC は全ページ合格水準・モバイルも致命的状態は脱した
